Access management

In order to optimize site performance, ensure that staff members and contacts are only granted the access rights they need, and that they are not assigned multiple security roles on the same items. We also recommend using staff and contact groups to assign roles to make it easier to track users and assign them the appropriate level of access.

Manage staff access

When granting access to staff members, you should consider the scope of access each staff member requires.

There are three levels of scope you can assign when you assign security roles: system-wide, entity-specific and item-specific. We recommend you keep in mind the following guidelines on role scope when you assign roles.

  • Where possible, assign entity-specific roles instead of system-wide roles.

    System-wide roles grant users access to all items across your Cloud site. If you have a large amount of data stored in your Cloud site, users with system-wide roles may experience slow loading times and decreased performance. In this case, you may want to consider granting entity-wide to users who do not require access to all items in your site to improve performance for those users. See Grant users and groups access to entities for more information.

  • Avoid assigning item-specific roles.

    Item-specific roles grant users access only on specific items, such as files or activities. Users with these roles will often experience slower performance compared to users with entity-wide roles, because in order to grant an item-specific role to a user, they must already have an entity-specific role for the entity the item is stored in. Granting users multiple roles on the same item often causes performance issues.

    Note that if you create confidential items within entities, users will require an item-specific role to access the item. We recommend you minimize the use of confidential items within entities to avoid performance issues.

Optimize role assignment

After you determine the level of access users require, you also need to ensure they aren't granted multiple roles on the same items.

When you assign roles to users, consider the following guidelines:

  • Assign only the role with the highest level of access the user requires.

    Note: The areas of overlap in this and the following diagrams indicate non-optimal role assignment.

    For example, it's not necessary to assign a user the Viewer, Editor, and Owner roles on a file. The Owner role grants the highest level of access, so only that role is necessary.

  • Do not assign users to groups with duplicate permissions.

    For example, staff members should not be assigned to two different staff groups that both grant the Owner role on the same set of entities or files.

  • Do not assign users system-wide, entity-specific, and item-specific roles on the same items.

    For example, if a user is assigned the Entity - Owner role for a specific entity, this role will give them Owner-level access to all items within the entity. It is not necessary to also assign item-specific roles to this user for all the items within the entity.

Restrict contact access

When granting access to contacts, ensure that you are only granting access to contacts who will need to log in to Cloud to respond to queries or file requests. If a contact does not need to log in, but you want to assign them to the entity to keep track of their contact information, you can assign them the Associated role on the entity. This role will ensure you can keep track of your contacts without granting them access to your Cloud site.

Best practices for Share Entity permission assignment

The Admin, Entities Admin and Owner built-in roles allow users to grant other staff or contacts access to an entity. In addition to these roles, you can also create custom roles with the Share permission for entities to assign users the ability to grant entity access. This custom role can be assigned to specific staff system-wide to manage all entities or at an entity level to manage specific entities.

Note that users that are assigned roles with the Share permission are able to grant themselves and others all available permissions on the entity. These users are not limited to the permissions they are currently assigned, which means they have the ability to perform actions such as granting unauthorized access to themselves or other users, and deleting entities. In order to ensure users are not taking unauthorized actions, you can monitor staff activity.

Monitor staff activity

You can use the Activities app to monitor staff activity. When permissions are assigned to any user, a system activity is generated and can be viewed in the Activities app. A system activity is also generated when entities are deleted. See Check system and file history to learn how to view system activities.

Subscribe to notification emails

In addition to using system activities to monitor activity, we also encourage the firm administrator to subscribe to events that may violate your company policy. For example, the firm admin can select to be notified whenever access to an entity is shared with a user and when an entity is deleted. See Change notification settings for more information.

The notification emails include links to view the system active details in Cloud. If you want more detailed information about the system activity to be included in the email, such as the entity name, you can enable Detailed Notifications in your email settings. See Customize system-generated emails for instructions.

Prevent deletion of entities

If you want to ensure entities cannot be deleted, a workaround is available with Caseware Cloud Time invoices to block the deletion of entities.

Caseware Cloud does not allow entities to be deleted when the entity has pending unpaid invoices. In order to prevent entity deletion, you can set up a placeholder unpaid invoice for the entity. This invoice will prevent all staff from deleting the entity, even if they have the necessary permissions. We also recommend granting access to the Time app only to your firm admins. This will prevent other staff members from closing the invoice so they can delete the entity.

Use staff groups to manage entity access

We recommend using staff and contact groups to assign roles to make it easier to track users and assign them the appropriate level of access. Groups can be especially useful if you need to manage access permissions for a large number of staff members across different regions. Note that staff members will only be able to view and access entities they are assigned to. When you create and assign staff groups by region, you can ensure your staff members only have access to entities in their own region. This section will outline how to organize your staff groups by region and assign them to entities.

Import entities and assign tags

We recommend tagging entities by region so you can easily filter your client entities to find the correct entities and assign staff groups to them. You can create and assign tags to entities when you import them into Cloud.

Cloud enables you to add multiple client entities in a single procedure using a .csv file or a block of delimited text.

When using either method, ensure that:

  • Each row represents one client entity, and
  • Each column represents a property, such as their name, start date, and tags. To assign multiple tags, place the tags between quote marks and separate them with commas. Any tags that do not currently exist in Cloud will be created by performing the import.

  • The dates in the Client Since and Inactive Date column must be in the same format as the date format set in your personal regional settings.

We recommend adding tags to your csv that match the security groups (Country/Region) so you can easily filter your client entities and improve the efficiency of assigning staff groups to them.

In this example, an administrator wants to add two client entities:

Name, Location, Entity Number, Client Since, Fiscal Year End, Tags

Chockablock Industries, Johannesburg, 4829, 09/07/2012, December, Africa

Superior Furniture, Singapore, 5937, 22 /08/2010, December, SIngapore

The entities are added to Cloud. Any existing entities with the same Entity Number are modified.

Create groups and assign staff

You can now create your staff groups and assign staff members to them.

To create a new group:

  1. Ensure that you have the Staff Admin role or equivalent privileges.

  2. From the Cloud Menu, select Groups.

  3. Select New, then choose Staff Group. The New Group dialog opens.

  4. Enter a Name and Description for your group.

  5. From the sidebar, select Members. Choose the users you want to add to this group.

  6. From the sidebar, select Apps. Choose the apps the users in the group should have access to. You may choose all apps, or you may want to limit the apps to those that are specific to the region the staff are working in.

    Note: For a staff member to be assigned to an engagement, they need to have access to the app the engagement is based on. If you want to ensure users in the group cannot create any engagements, deselect the Create checkbox for the apps you're sharing.

  7. Select Save.

The new group is created. You can see the new group on the Groups page.

Grant entity access to groups

The final step is to assign the groups you have created to the necessary entities. To make it easier to find the entities you require, you can use the region tags you created earlier to filter the entity list.

To grant entity access to groups:

  1. Ensure that you have the Owner role or the equivalent privileges for the appropriate entities.

  2. From the Cloud menu, select Entities.

  3. Select Filter () in the search bar.

  4. Under Tags, select the tag for the appropriate region.

    The entity list will now only display entities in the region you have selected, so you can easily assign your groups to the correct entities.

  5. Select the checkbox in the title bar to select all the entities in the filtered list.

  6. Select More Actions | Share.

  7. Select Staff.

  8. Choose the appropriate role from the drop-down .

  9. In the search field, enter the name of the group you want to grant access the role to.

  10. Select Share.

The selected staff groups now have access to the entities.

Grant access to a specific entity

There will be situations where you want to assign a specific person to an entity or engagement separate from the security groups you have created. This would occur when you need to share an engagement from one country with a user that is based in a different country.

To grant access to an entity:

  1. Ensure that you have the Owner role or the equivalent privileges for the entity.

  2. Entities.

  3. Select the entity you want to grant access to, then select Share ().

  4. Search for the groups you want to grant access to, then select Assign Roles to (#).

  5. Choose the appropriate staff role.

    Roles that users already have due to previous settings appear under Inherited.

  6. Select Share.

You can also grant access to multiple entities simultaneously by selecting more than one entity. Note that if you grant access to multiple entities simultaneously, you can only assign one role for each group you select. For example, if you select entity A and entity B, then you can only assign the selected staff member the Viewer role on these two entities. You cannot assign Viewer on one entity and Editor on the other, for instance.

Add or remove users from a group

After groups are created, you can manage the membership of the groups by adding or removing users as appropriate.

The group's membership is updated. To see the current membership of the group, select it from the Groups page.

Create country/regional tags manually

We recommend tagging clients by Country/Region so you can easily filter your client entities to find the correct entities and assign staff groups to them. If you did not create tags automatically during the client csv import, as shown in step 2, or if you need to add more tags after import, you can also create them manually.

To manually create a tag for a region:

  1. Ensure you have the Settings Admin role or equivalent privileges.

  2. From the Cloud menu, select Settings.

  3. Select Customization | Tag Management.

  4. Select Add to open the Add Tag dialog.

  5. Enter a Name and specify Entities from the Tag Type drop-down.

    Note: We recommend using the same names as the country/regions used to define the security grouping

    .

  6. Select OK.

Assign tags to entities (clients) manually

You can manually assign tags to an entity, if necessary.

To manually assign a tag to an entity:

  1. Ensure you have the Entities Admin role or equivalent permissions.

  2. From the Cloud menu, select Entities.

  3. Select the entities you want to tag for a specific region from the list.

  4. Select Tag ().

  5. Select the appropriate tag from the Unassigned tags to assign them to the entities. You can also search for tags in the search box.
