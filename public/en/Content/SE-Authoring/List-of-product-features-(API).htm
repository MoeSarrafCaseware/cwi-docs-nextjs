List of product features (API)

The following list describes the product features that you can enable/disable in your SE product.

See How to author your product template to learn more about how to author your SE product template.

Additional entity fields

Define the customFields field to customize the fields in the Create Entity dialog.

Additional annotations

You can include the com.caseware.reference module in the product.json file to enable additional annotation types.

Analytics Hub

The Analytics Hub is a centralized workspace within your engagement environment that allows teams to view, manage, and execute analytic tests directly linked to checklist procedures. To learn more, refer to Manage analytic tests in the Analytics Hub.

To enable this feature, in Product Settings, go to Analytics Hub, select General | Enable Analytics Hub.

Areas

Allows you to summarize financial statement groups at a higher level for risk identification and assessment. For more information, see Add areas and cycles.

Assertions

Include the com.caseware.assertion module in the product.json to enable annotations in your Cloud app.

Automapping

Automatically maps sections of data files on import. To learn more, see Set up automapping.

See the importing and automapping SDK tutorial for more information on the automap property.

Automatic rounding

To enable automatic rounding in the financial statements, create a script called financial-rounding-setup.js and include it in the “js” array of your product.json file. In this script, include the following:

(function() {
     ‘use strict';
     wpw.financials.financialsSetup.roundingErrorGroupNumbers = ['A', 'B'];
})();

‘A’ and ‘B’ are strings representing the group numbers of the groups in the structure that should sum to zero. If required, you can include more than two groups.

For example, if the grouping structure contains two groups Assets and Liabilities that should sum to zero, then replace the ‘A’ string in the above function with the group number for Assets and replace the ‘B’ with the group number for Liabilities. Liabilities and Assets are then summed and checked for rounding errors.

Note that rounding differences are applied by default to the first non-zero account in the lowest child level that appears in the first group in the trial balance. For more information, see Change the default group for rounding differences.

Carry forward

The carry forward process enables users to reuse information from the prior year when creating an engagement. To enable carry forward in your product, refer to Enable carry forward. To learn more about the user experience for carry forward, refer to Carry forward an engagement.

Copy content from other engagements—risks, controls or checklists

Users can copy risks, controls or checklists from an engagement, or from a component within a group audit, into their current engagement. To learn more, refer to Copy risks, controls or checklists from other engagements.

To enable this feature, in Product Settings, go to Features | Content Copy, and then select Enable Content Copy.

Checklist

Enables the checklist functionality.

For more information, see Add a checklist.

Contacts control

This property controls contacts' access to engagement. The default setting is that contacts cannot edit or view any engagement. To grant view access to contacts assigned to the engagement, set contactAccess to READ.

Note that once you grant contacts access to engagements, you cannot limit what areas of the engagement file they can access. We recommend using this feature for reporting products only and not audit products. We also recommend thoroughly reviewing the feature before releasing your product to ensure it works as you expect.

Cycles (business cycles)

Used to further organize the groups in the Trial Balance.

For more information, see Add areas and cycles.

Detailed Documents view

Makes the Detailed View in the Documents page the default view.

Dimensions

Dimensions are properties that can be assigned to accounts in an engagement. Each account can be assigned one or more dimensions. To enable dimensions in your product, in your product.json file, in the features array, set both dimensions and trialBalanceDimensions to true.

"feature": {
    "dimensions": true,
    "trialBalanceDimensions": true,

For more information, see Add dimensions.

Engagement from Source

To allow users to create new engagements based on existing files, you must set the property this.canCreateEngagementFromSource = true in the collaborate.js file for your product.

For more information, see Create a new engagement based on an existing file.

Engagement Properties Prompt

Display the Engagement Properties prompt each time users open a new engagement. For more information, see Set the engagement properties.

Engagement Search

For more information, see Locate and access documents in the engagement files.

Envision

Enables the ability to add the out-of-the-box Envision data profile to your product. Envision data profiles are dashboards that display information about imported transactions in engagements. For more information, see Envision.

To enable Envision, add "com.caseware.an.e.analytics.ai" to the set array and set the property "envisionProfile": true.

You can then use the Add Document icon () in the Documents page to add the Envision Profile custom form to your product.

External life cycle

Set the externalLifecycle property to true if you want the engagements in your SE product to be created or deleted outside of Caseware Cloud.

Firm customizations

Allows firm authors to customize the template used to create new engagements. This property is enabled by default.

Group audit

The Group Audit feature streamlines the use of common work areas within a group audit. This includes the ability to:

  • Identify common work areas for various group audit components.

  • Share common work areas with component auditors

  • Adapt common work areas to engagements with evolving statutory standards and complex requirements

To enable this feature, select Group Audit from the Features list in Product Settings and click Enable Group Audit.

If both Consolidation and Group Audit are enabled in a product, you will also be able to choose the default structure type of either Consolidation or Group Audit. This determines what the user will see first in the options menu when converting a single engagement into a group audit or consolidation.

Help

Allows you to provide your users with a link to a help page, such as your product information site.

Imports

 

The imports array in the product.json file defines which data connectors are available in your product, enabling users to import client data. A data connector is a component designed to extract data from a specific provider. A provider can be third-party accounting software, a particular file format (such as CSV), or even another engagement. Provider binding and import binding are synonymous terms used to describe this integration.

To enable import options, add the desired data connectors to the imports array. To learn more, see Enable Provider Bindings imports. These options appear to end users on the Data page.

You can also build a custom integration to import data from online accounting platforms. To learn more, see Provider Bindings SDK.

Import datasets

To support imports of general ledger data, subledger transaction data, and trial balance data from CSV or Excel files, you can configure which dataset types are available to users.

To do this, include the relevant schemaId values in the product.json file, as shown in the code sample below. Remove any schemaIds for dataset types you do not want to expose to users.

Refer to the linked descriptions below for details on each dataset type. To learn more about how users import data into their engagements, refer to Import and manage data.

"importTaxonomy": {
					"id": "cwi.taxonomy.transactions",
					"version": "0",
					"schemaIds": [
					"cwi.schema.general_ledger",
					"cwi.schema.ap_invoices",
					"cwi.schema.ap_payments_made",
					"cwi.schema.ap_suppliers",
					"cwi.schema.ap_open_balances",
					"cwi.schema.ar_invoices",
					"cwi.schema.ar_cash_received",
					"cwi.schema.ar_customers",
					"cwi.schema.ar_open_balances",
					"cwi.schema.inventory_transactions",
					"cwi.schema.inventory_on_hand"
					]
			}

To enable users to import data that does not conform to any dataset type, you can add "schemaless": true to the "feature" section in the product.json file. As a result, users will have the option of selecting the Other dataset type.

Inherent risk factors

As primary or enterprise authors, you can enable Inherent risk factors from your Product Settings under Risk. This area allows you to name and categorize inherent risk factors for your users to select from when assessing a risk.

The following are category option for inherent risk factors:

  • Complexity

  • Subjectivity

  • Change

  • Uncertainty

  • Susceptibility of misstatement due to management bias

  • Susceptibility of misstatement due to fraud risk factors

  • Other

Category names can be customized. Subcategories can be created and customized under each Category. Category options can be selected or deselected as options for your users.

To enable Inherent Risk Factor

  1. From the product template, click the Information icon[] at the top of the screen.

  2. Click the Settings option to open Product Settings.

  3. Click Fields under the Risk section

  4. Click the Inherent Risk Factors toggle to enable or disable the feature

  5. Customize the names of the risk factor categories as needed

  6. Add and name sub-categories using the Add icon [] under each category as needed.

  7. Under Display, you can select where the Inherent Risk Factors feature will display in an engagement: in the Risk Card, when creating a new risk, in the Risk dialog box, or both.

The selection options will become available to your users in the Risk card when creating a New Risk and the Risk dialog box, depending on what has enabled for your users.

Final rounded balances for Current Year and Prior Year

On the Accounts tab on the Data page, users can choose to view final rounded balances for the Current Year (CY-Rounded) and Prior Year (PY-Rounded) by selecting these columns in the Columns dropdown, To learn more, refer to View accounts. The balances in CY-Rounded and PY-Rounded will match what is shown in the Financial Statement. This feature ensures alignment between your data and financial statements.

To enable this feature, in Product Settings, select Trial Balance | Data Page | Enable rounded columns.

The rounding conditions (for example, round by thousands vs millions) can be set from Product Settings under Financial Statements | Enable rounding in the financial statements.

Financial Statements as an option in the Create a document list

On the Documents page, users can add documents by selecting the Create a document icon () and then selecting the type of document they want to add to their engagement.

In product.json there's an option in the feature array that can be included "statement": true. Setting this to false (or not including it) removes the Financial Statement option from the Create a document list.

Risk assessment per risk

As a primary or enterprise author, you can enable your users to perform a risk assessment on an individual risk. This includes the same level of assessment that users would apply to the assessment of risk for the entire engagement. The individual assessment returns values beside all the tracking metrics.

This feature supports the user when a greater level of documentation is required to meet the user’s company or industry standards. This feature provides a place to record individual risk assessment levels.

Enable Risk Assessment per Risk

  1. From the product template, click the [] Information button at the top of the screen.

  2. Click Settings to open Product Settings.

  3. Click Fields under the Risk section

  4. Click the Assessment of Specific Risks toggle to enable or disable the feature

  5. Customize the name of the feature as it will appear in your users’ engagements.

Once enabled, your users can see options to individually assess each risk based on a scorecards measuring the following:

  • IR - inherent risk

  • CR - control risk

  • LMO - Likelihood of Misstatement Occurring

  • MPM - Magnitude of Potential Misstatement

Inherent risk (IR) is automatically calculated based on your selections of Likelihood of Misstatement Occurring (LMO) and Magnitude of Potential Misstatement (MPM) and the CR.

RMM is automatically calculated based on IR and CR assessments. RMM drives the visibility of procedures in the Risk response phase.

Assessment within the individual risk

Your users can also view the assessment of the individual risk in the overall risk assessment from the Assessment tab.

Forms and Form sets

To include the risk assessments, risks, controls, materiality and misstatements reports in your Cloud app, include the general form set com.caseware.ca.general in the set property. You must also enable the following:

"feature": [ "cycles", "riskAssessment" ]

"module": [ "com.caseware.visualization.charts", "com.caseware.assertion" ]

If you don't want to include all the reports, you can include specific reports. Note that if you choose to include specific forms, you must enable the required property or module. For example, to enable the Controls report:

  • Include the com.caseware.ca.general.control-report in the form property

  • Enable the Cycles property

iXBRL

Enables Inline eXtensible Business Reporting Language (iXBRL) functionality throughout the product. iXBRL is an open standard that allows firms to prepare financial statements in a format that provides the structured data that regulators and analysts require, while allowing preparers to retain full control over the layout and presentation of their report.

A conversion support file and a pointer to it must be added in the API. See the XBRL SDK tutorial for the detailed instructions.

If you have at least one XBRL taxonomy set up, XBRL review mode is available to end users and enabled by default. If you want to hide an XBRL taxonomy from end users, you can set the "hideFromReviewMode" property to true in the XBRLTaxonomyDefinition.

Note that if all taxonomies are hidden, XBRL review mode is not available to end users.
